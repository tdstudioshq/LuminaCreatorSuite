Project overview

▶

CABANA is a premium creator OS — link-in-bio, storefront, media kit, first-party analytics — being evolved additively into a creator subscription platform (members, follows, posts, fan subscriptions, messaging, tips, payouts). Product name in code and UI is CABANA; the repo folder is LuminaCreatorSuite.

The guiding rule: preserve all existing creator functionality. Each phase adds tables + RLS and a thin, RLS-scoped server-action tier in strict dependency order, and never moves real money or touches the remote Supabase without explicit, gated approval.

Architecture summary

▶
  • Routing/SSR: file-based TanStack Router in src/routes/ (flat dot-notation); SSR entry src/server.ts wraps the handler with a branded error page; routeTree.gen.ts is generated — never hand-edit.
  • Three Supabase trust boundaries: supabase (anon browser client, RLS-enforced) · supabaseAdmin (service role, server-only) · requireSupabaseAuth middleware (per-request RLS-scoped client). A fourth, optionalSupabaseAuth, serves guest-callable reads (feed, discovery).
  • T2 protected actions: every createServerFn composes client attachSupabaseToken + server requireSupabaseAuth, so handlers run under the caller's RLS — never the service role. Handlers stay thin; behavior lives in pure, repository-injected modules.
  • Pure business layer: 16 cabana-* modules (money, entitlements, posts, messaging, moderation, payouts, analytics…) unit-tested without React, Supabase, or a browser — this is where new domain logic must go.
  • Event generation at the DB layer: notifications and audit rows are written by SECURITY DEFINER triggers (emit_notification, report/payout audit triggers), so events fire uniformly for every write path.
  • Design system: luxury dark / glass / iridescent utilities in src/styles.css (Tailwind 4, CSS-config); shadcn/ui primitives in components/ui/; social surfaces render inside the three-column SocialShell.

Auth / roles / RLS notes

▶
  • Four principals: guest (public reads only) · member (/account home) · creator (/dashboard, default on signup) · staff (admin or moderator via user_roles). is_current_user_admin() / is_current_user_staff() wrap the revoked has_role for policies.
  • Signup branching: the handle_new_user trigger reads raw_user_meta_data.account_type; only the exact string "member" opts out of creator provisioning.
  • Route guards are client-side only (redirects in dashboard.tsx/account.tsx). Data stays safe because every server action independently validates the bearer token — but guarded shells can flash for signed-out users.
  • Writes go through RPCs or column-scoped policies: fan subscriptions, conversations, purchases, tips, payout reviews are SECURITY DEFINER RPC-only; clients may only flip notifications.read_at.
  • Identity safety: cross-account discovery uses ID-free views/RPCs. Known exception: the legacy public read of creator_profiles still exposes user_id (see risks).
